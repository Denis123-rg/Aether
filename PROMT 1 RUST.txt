General Guidelines
Do not modify the production code except if absolutely necessary to expose internal functionality for testing (e.g., add #[cfg(test)] helpers, or make some items pub(crate)). Prefer testing through public interfaces.

Write idiomatic Rust tests using the built‑in #[test] attribute. Use assert_eq!, assert!, assert_matches!, and where appropriate, use proptest for property‑based testing or mockall for mocking external dependencies.

Leverage existing test infrastructure – many crates already have test modules; add new tests there or create new test files under tests/ if integration tests are needed.

Focus on coverage gaps: use the coverage report to identify specific functions, branches, and error paths that are not exercised. Write tests that trigger those paths.

Ensure all new tests pass – they must compile and run successfully without introducing failures in existing tests.

Keep tests realistic – use real‑world‑like inputs (e.g., actual Ethereum addresses, pool data, simulation parameters) but avoid relying on external network calls unless absolutely necessary; use mocks or in‑memory stubs.

Sub‑agent coordination: Assign each file to a sub‑agent. Each sub‑agent is responsible for that file’s test coverage. They should communicate to avoid duplication and ensure consistent testing style.
