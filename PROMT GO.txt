General Guidelines for Writing Tests
Do not alter production code unless absolutely necessary to expose internals (e.g., add //export or test-only helpers). Prefer testing through public interfaces.

Use table‑driven tests to cover multiple scenarios concisely.

Mock external dependencies (RPC, databases, signers, HTTP) using interfaces and test doubles. Use gomock, testify/mock, or write manual stubs.

For main() functions (0% coverage), refactor the logic into a separate function (e.g., runMain() or realMain()) that accepts arguments (like io.Writer, context, config) and test that.

For generated ProtoMessage methods, test by creating a message and calling ProtoReflect().Descriptor() or use the generated code in an integration test that marshals/unmarshals.

For no‑op methods (e.g., Record, Close on NoopMetricsStore), test that they return without error and do nothing.

Cover error paths – use mocks to simulate failures (DB errors, network timeouts, invalid inputs).

Keep tests realistic – use actual‑looking addresses, hashes, and configuration values.

Ensure no existing tests break – all tests must pass.

Specific Advice per Package
cmd/executor: Use httptest for admin endpoints, grpc_test for remote signer, and mock clients for Flashbots.

internal/db: Use an in‑memory SQLite or test‑containers for Postgres, but for no‑op methods just verify they return.

internal/events: Mock Redis or use in‑memory pub/sub.

internal/grpc: Test DialWithOptions with invalid addresses, TLS misconfigurations.

internal/risk: Test state transitions, circuit breaker, daily reset with mocked time.

internal/signer: Test encryption/decryption with known key pairs, file I/O errors.

internal/strategy: Test score and Pick with various A/B test data.
