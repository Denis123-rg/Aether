Phase 1: Verify Build and Test Suite
Before writing new tests, ensure the entire project builds and all existing tests pass. Run:

bash
go mod tidy
go test ./... -v
If any tests fail, fix them (but do not change production logic unless absolutely necessary). Once everything is green, proceed to Phase 2.

Phase 2: Write Tests to Achieve ≥95% Coverage
For each function in the list, you must write tests that cover all paths, including:

Success paths – normal operation.

Error paths – invalid inputs, network failures, database errors, timeouts, etc.

Edge cases – empty inputs, zero values, boundary conditions.

General Guidelines
Do not alter production code unless you need to expose internal functions for testing (e.g., add a test‑only helper or make a function public for testing). Prefer testing through exported APIs.

Use table‑driven tests to cover multiple scenarios concisely.

Mock external dependencies (RPC, databases, filesystem, HTTP clients) using interfaces and test doubles. Use gomock, testify/mock, or manual stubs.

For main() functions (0% coverage): refactor the logic into a separate function that accepts arguments (like io.Writer, context, config) and test that function. Keep the actual main() minimal.

For generated ProtoMessage methods: test by creating a message and calling ProtoReflect().Descriptor() or by using the generated code in an integration test that marshals/unmarshals a message.

For no‑op methods (e.g., Record, Close on NoopMetricsStore): test that they return without error and do nothing.

Ensure all tests pass – run go test ./... repeatedly and fix any failures.

Specific Advice per Package
deploy/docker/mock-builder
main() is trivial. Refactor into a run() function that can be called with a mock context. Test that it exits correctly and handles missing arguments.

internal/db/ledger.go (no‑op methods)
These are methods on NoopLedger. Write tests that create a NoopLedger and call these methods, asserting they return zero values and don't panic.

internal/db/timescale.go (no‑op methods)
Similar to above – test NoopMetricsStore methods.

For ProtoMessage – create an instance of each message type, call ProtoReflect(), and check it returns a non‑nil descriptor. This exercises the method.

For mustEmbedUnimplemented* and testEmbeddedByValue – they are used for forward compatibility; you can simply create a server struct that embeds them and ensure it compiles. But to cover, you can call these methods (they are no‑ops) in a test.
