General Guidelines for Writing Tests
Do not change production code unless necessary to expose internals (e.g., make a function pub(crate) or add #[cfg(test)] helpers). Prefer testing through public interfaces.

Use the existing test infrastructure – add new tests to existing #[cfg(test)] modules or create new test files under tests/ for integration tests.

Write idiomatic Rust tests using assert_eq!, assert!, assert_matches!. Use proptest for property‑based testing where appropriate.

Mock external dependencies (RPC, database, file system) using mockall, httpmock, or custom trait implementations.

Focus on uncovered branches – use the coverage report to identify specific lines, functions, and error paths that are not exercised.

For binary files (aether_replay.rs, aether_profit_scorer.rs, main.rs):
Refactor the logic into a separate function that can be unit‑tested. For example, move the main logic into a run() function that accepts parameters and returns a Result, then test that.

For tracing_init.rs – test initialisation with different environment variables, both success and error paths.

For fee_on_transfer.rs – many edge cases (different fee percentages, honeypot detection) – use property‑based tests.

All new tests must pass – they must not break existing tests.
