General Guidelines for Writing Tests
Do not modify production code unless absolutely necessary to expose internals (e.g., make a function pub(crate) or add #[cfg(test)] helpers). Prefer testing through public interfaces.

Use the existing test infrastructure – add new tests to existing #[cfg(test)] modules or create new integration test files under tests/ when needed.

Write idiomatic Rust tests using assert_eq!, assert!, assert_matches!, and where appropriate, use proptest for property‑based testing.

Mock external dependencies (RPC, databases, file system) using mockall, httpmock, or custom trait implementations. Do not rely on external network calls unless absolutely necessary (use mocks).

Focus on uncovered branches – use the coverage report to identify specific lines, functions, and error paths that are not exercised.

For binary files (aether_replay.rs, aether_profit_scorer.rs, main.rs): refactor the main logic into a separate function (e.g., run()) that can be unit‑tested, then test that function with mocked dependencies.

For tracing_init.rs: test initialization with different environment variables, both success and error paths.

For fee_on_transfer.rs: many edge cases (different fee percentages, honeypot detection) – use property‑based tests or exhaustive table‑driven tests.

All new tests must pass – they must not break existing tests.
